## 1. 策略选择

| 策略 | 说明 | 推荐度 |
|------|------|:------:|
| A. 从零开发 | 完全自主实现所有层次 | ❌ 不推荐 |

**选择策略 B 的理由**：
1. 核心链路已通（IPC + WebView + 自定义协议），无需重新验证
2. 代码基础扎实（TAO 1281 行 + WRY 279 行 + Tauri Core 大量改动）
3. 工作量减少 55-60%（从 90-165 人天降至 37-73 人天）
4. 风险大幅降低（最大风险从"WRY 从零实现"变为"PC 适配"）

## 3. 阶段 1：复现 Demo（1-2 周）

### 目标

在本地环境中成功运行 tauri-demo，理解完整架构和构建流程。

### 任务清单

| # | 任务 | 预估 | 产出 |
|---|------|:----:|------|
| 1.1 | 安装 DevEco Studio + HarmonyOS SDK/NDK | 0.5天 | 开发环境就绪 |
| 1.2 | 配置 Rust OHOS 交叉编译工具链 | 0.5天 | `cargo build --target aarch64-unknown-linux-ohos` 可用 |
| 1.3 | 克隆 tauri-demo 并安装依赖 | 0.5天 | 项目可编译 |
| 1.4 | 执行 `cargo tauri ohos build` | 0.5天 | 生成 HAP 包 |
| 1.5 | 在 DevEco Studio 中运行到设备/模拟器 | 0.5天 | Demo 在设备上运行 |
| 1.6 | 阅读源码，理解 TAO/WRY/Tauri Core 的 OHOS 实现 | 2天 | 架构理解文档 |
| 1.7 | 记录环境搭建过程，编写开发指南 | 1天 | 开发环境文档 |

### 操作步骤

```bash
# 1. 安装 Rust OHOS target
rustup target add aarch64-unknown-linux-ohos

# 2. 安装 Tauri CLI (feat/open-harmony 分支)
cargo install tauri-cli --git https://github.com/tauri-apps/tauri --branch feat/open-harmony

# 3. 安装 ohrs
cargo install ohrs

# 4. 克隆 demo
git clone https://github.com/richerfu/tauri-demo.git
cd tauri-demo

# 5. 安装前端依赖
pnpm install

# 6. 构建
cd src-tauri && cargo tauri ohos build

# 7. 在 DevEco Studio 中打开 src-tauri/gen/ohos 并运行
```

## 8. 关键决策点

### 决策 1：是否维护独立 fork？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **维护独立 fork** | 完全控制，快速迭代 | 需要持续与上游同步 |
| 提交 PR 到官方 | 长期可持续 | 审查周期长，可能被拒 |
| **推荐**：先独立 fork 开发，成熟后提交 PR | | |

### 决策 2：openharmony-ability 的依赖策略

| 选项 | 优点 | 缺点 |
|------|------|------|
| 直接依赖 | 简单，利用社区维护 | 受制于第三方更新节奏 |
| **Fork 并自行维护** | 完全控制，可快速修 bug | 维护成本 |
| **推荐**：先直接依赖，遇到阻塞时 fork | | |

### 决策 3：PC 适配的优先级

| 选项 | 优点 | 缺点 |
|------|------|------|
| **先手机后 PC** | 快速出成果，验证核心功能 | PC 适配延后 |
| 直接做 PC | 一步到位 | 周期长，风险高 |
| **推荐**：先手机/平板，再 PC | | |

## 9. 风险缓解计划

| 风险 | 缓解措施 | 触发条件 |
|------|---------|---------|
| fork 与上游不同步 | 每 2 周 rebase 一次 | 上游有重大更新 |
| openharmony-ability 有 bug | 准备 fork，直接修复 | 阻塞开发 > 2 天 |
| PC API 与手机差异大 | 尽早获取 2in1 真机测试 | 阶段 2 结束前 |
| 构建工具链不稳定 | 记录所有环境配置，编写自动化脚本 | 新成员无法搭建环境 |
| 性能不达标 | 分析 IPC 链路瓶颈，优化关键路径 | 启动时间 > 3 秒 |
